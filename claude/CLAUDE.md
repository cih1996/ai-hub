§1 全局规则定位（仅流程骨架）

全局规则只做三件事：
1. 定义执行顺序。
2. 定义读写触发条件。
3. 定义安全边界。

具体业务策略、角色职责、项目细节一律下沉到：
- 团队规则：`~/.ai-hub/teams/<group_name>/rules/*.md`
- 会话规则：session-level system prompt
- Skills：操作手册与工具约束

---

§2 固定执行顺序（必须）

每次收到消息按以下顺序执行：
1. 分流：P0 会话维持 / P1 执行任务 / P2 复盘沉淀。
2. 锁定：确认任务对象、目标范围、前置约束。
3. 读取：会话规则 → 团队记忆/知识 → 全局记忆/知识 → 相关 Skill。
4. 执行：能走 Skill 的动作必须先走 Skill。
5. 回传：先结论，后证据。
6. 沉淀：按触发条件写入记忆/知识/规则改进。

---

§3 读取与写入触发

必须读取：
- 命令执行、配置变更、发布、调度、自动化脚本调用。
- 记忆库/知识库任何写入前，必须先检索同主题现有记录。

可跳过读取：
- 纯问候、确认类短回复。

必须写入：
- 错误完成根因定位。
- 形成可复用方法并完成验证。
- 规则被纠错或执行顺序被调整。
- 上下文压缩前关键状态快照。

写入原则：
- 先检索后写入，命中则更新，避免重复。
- 团队私有内容写团队库；跨团队复用内容写全局库。

§3.1 记忆库/知识库治理（强制）

写入前（必须全部满足）：
1. 先检索 memory 与 knowledge 的同主题记录。
2. 判断是否已有“主文件”（canonical file）。
3. 命中主文件时只更新，不新建平行文件。

写入时（统一规范）：
1. 一主题一主文件：同一主题长期维护同一个文件。
2. 采用“事实化表述”，不要写“本次新增/增强版/本次修改”等时态词作为正文主体。
3. 以“当前有效状态”为主，历史变更写到同文件末尾“变更记录”区，不拆分成多个重复文件。
4. 文件名稳定且可复用，禁止为同主题反复创建 `xxx-新版.md`、`xxx-增强版.md`。
5. 禁止过程叙事：不要写“之前发生了什么/为什么当时做不到/本次如何解决”，仅保留最终有效方案与必要边界。

冲突处理：
1. 多文件内容冲突时，先选定一个主文件。
2. 其他文件并入主文件后标记为废弃或删除，避免多份真相并存。
3. 未完成主文件收敛前，不继续扩写新文件。

§4 多会话最小防错

- 执行类调度必须带上下文头：`[group_name|scope|target|task_id]`。
- 子会话回调必须带同一 `task_id`。
- 未锁定 scope/target 的执行任务不得下发。

---

§5 Skill 优先（重点）

- Skill 是执行协议，不是参考文档。
- 发现“执行不流畅、重复步骤、规则不足”时，优先补充/修订 Skill，再继续。
- 全局规则不承载细节动作，细节统一沉淀在 Skill。

---

§6 安全边界

- 禁止猜测 API、参数、命令。
- 上下文恢复必须使用已知接口（skill），禁止臆造路径
- 不确定先查证；高风险操作先确认。
- 批量任务执行到可验证节点再汇报，避免空转与误报。

---

§7 自我纠错机制（强制）

**触发条件**（满足任一即触发）：
1. AI 按规则/知识库执行，但结果出错
2. AI 自主排查后找到与规则/知识库不符的正确方法
3. 用户提供正确思路，AI 按新思路执行成功

**纠错流程**（按顺序执行）：
1. **识别缺口**：判断是规则缺失、规则错误、知识库缺失还是记忆库缺失
2. **验证正确方法**：执行验证，确认新方法有效后再写入
3. **立即沉淀**：
   - 规则错误/缺失 → 通过「规则管理」Skill 更新规则
   - 知识库缺失 → 通过「向量知识库」Skill 写入知识
   - 记忆库缺失 → 通过「向量知识库」Skill 写入记忆
4. **写纠错记忆**：文件名格式 `纠错-<任务类型>-<日期>.md`，内容包含：
   - 问题描述 / 错误原因 / 正确方法 / 影响范围

**任务前纠错专项搜索（必须）**：
- 每次接到新任务，除常规知识搜索外，**必须额外搜索一次**：关键词 = `纠错 + 任务类型`（如 `纠错 向量搜索`、`纠错 会话创建`）
- 命中纠错记录时，**必须完整 Read 后再开始工作**，不得重复踩坑
